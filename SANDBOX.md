# 沙盒事件系统设计

## 概述

随行的沙盒系统模拟真实世界中不断变化的环境状态。不同于简单的"随机抽一个事件"，我们设计了分层事件架构，包含**常规事件、大型城市事件、剧情链、记忆彩蛋**四个层次，体现事件的复杂性、趣味性和个性化。

## 事件分层架构

```
沙盒事件
├── 第一层: 常规事件 (每tick持续)
│   ├── 天气变化 (每5 tick, 含6时段预报)
│   ├── 餐厅排队波动 (12家, 10%特殊事件)
│   ├── 场馆状态变化 (3%异常概率)
│   ├── 活动票源变化 (5%退票概率)
│   ├── 单车站库存变化
│   └── 生活事件 (45%/tick, 20+模板)
│
├── 第二层: 大型城市事件 (10%/tick, 冷却机制)
│   ├── 望京半程马拉松 → 封路+无网约车+单车满员+外卖延时+观赛特餐
│   ├── 798户外音乐节 → 道路管制+展览排队+单车骑光+半价票+啤酒套餐
│   └── 美团骑士节 → 全城免配送费+出餐延时+特价套餐+买一送一
│
├── 第三层: 剧情链 (特定事件触发)
│   ├── 失物招领: 吃完火锅 → 工卡落下 → 单车爆胎 → 会议迟到
│   ├── 连环加班: 老板加班 → 隔壁缺人 → 加班餐差 → 同事吐槽
│   ├── 惊喜变惊吓: 周末计划 → 女友约同一天 → 朋友饭局 → 两边难推
│   └── 天气陷阱: 户外计划 → 天气突变 → 风力加大 → 影院特价票
│
└── 第四层: 记忆彩蛋 (用户画像条件触发)
    ├── 去过3次川菜 → 隐藏菜单: 限量藤椒鱼火锅
    ├── 喜欢骑行 → 秘密越野路线探路名额
    ├── 喜欢悬疑/科幻 → 超前点映场
    └── 预算充足 → 黑珍珠餐厅周体验套餐
```

## 三级通知分类

| 类别 | 图标 | 推送规则 | 去重 |
|------|------|---------|------|
| 🔴 个人事务 | 红 | 一律提醒, 不分场景 | LLM判断相同话题→SILENT |
| 🟡 环境变化 | 橙 | 仅在用户确认了具体计划时提醒 | 关键词重叠≥2→跳过 |
| 🟢 机遇事件 | 绿 | 稀有度节流 + 6类独立开关 | RarityTracker频次控制 |

## 机遇事件子分类开关

用户点击手机顶部的🦉头像可配置:

| 开关 | 默认 | 涵盖事件 |
|------|------|---------|
| 🎫 演出门票 | 开 | 演唱会/脱口秀退票 |
| 🎬 电影票 | 开 | IMAX退票/点映 |
| 🖼 展览门票 | 开 | 艺术展/博物馆票 |
| 🍲 餐厅空桌 | 开 | 热门餐厅放桌/恢复营业 |
| 💰 特价优惠 | 关 | 今日特价/优惠券/会员权益 |
| ⚡ 限时活动 | 关 | 市集最后一天/群活动 |

## 频次控制

```
RarityTracker:
  urgent → 每次都推 (周杰伦退票)
  high   → 每次都推
  medium → 每2次推1次
  rare   → 首次必推, 之后每3次推1次
  common → 首次必推, 之后每5次推1次
```

## 数据一致性

所有组件共享同一份 mock 状态:

```
_weather_state (单例)
  ├── tick_weather() 写入
  ├── tick_venues() 写入 (场地因天气关闭时同步)
  ├── get_weather_state() 读取 → /api/weather → LLM工具
  └── ProactiveBrain 读取

_restaurant_state (单例)
  ├── tick_restaurants() 写入
  └── /api/restaurants 读取

_activity_state (单例)
  ├── tick_activities() 写入
  └── /api/activities 读取
```

## Mock 数据规模

- 12 家餐厅 (川菜/火锅/日料/北京菜/湘菜/粤菜/烧烤等)
- 8 个活动 (电影/展览/户外/运动/演出/市集/密室)
- 6 个预报时段 (afternoon/evening/night/tomorrow×3/saturday×3/sunday×3)
- 20+ 生活事件模板
- 3 个大型城市事件
- 4 个剧情链
- 4 个记忆彩蛋
- 10+ 场馆状态 (公园/影院/地铁/单车/健身房/美术馆等)
